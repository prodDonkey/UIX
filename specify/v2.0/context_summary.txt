3) 关键结论（本次会话共识）
- 仅嵌入 android-playground 不能天然展示“当前 runId 的结构化步骤”，因为它与 ui_demo 的 run 执行链路是解耦的。
- 若要严格 runId 绑定步骤，必须引入“可查询进度的执行层”（推荐独立 Runner 服务）。

4) 已发生的 UI 调整（ui_demo frontend）
- RunDetail 页面已嵌入 Android Playground iframe。
- 模块位置已按用户反馈互换：左侧主区域展示设备画面，右侧保留历史与实时日志。
- 相关文件：
  - /Users/yaohongliang/work/liuyao/AI/UI/ui_demo/frontend/src/pages/RunDetail.vue

5) 当前实现痛点（待 v2.0 解决）
- backend/app/services/run_service.py 仍以 subprocess 执行 npx midscene，步骤进度不可结构化查询。
- 前端当前可见实时日志，但“当前步骤”仅能依赖日志文本，无法稳定准确。

6) 建议下一步（进入实现阶段）
- M1：先落地 runner-service 最小可用接口
  - POST /runs/start
  - GET /runs/:runId/progress
  - POST /runs/:runId/cancel
  - GET /runs/:runId/result
- M2：ui_demo backend 接入 runner-service，替换现有 subprocess 执行链
- M3：前端 RunDetail 增加 /api/runs/{id}/progress 展示（当前任务/动作/完成度）
- M4：回归、灰度与 Runner 发布回滚预案验证

7) 配置建议
