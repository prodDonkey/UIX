# 可视化 YAML 脚本平台需求文档（V1.0）

## 1. 项目背景
当前团队已经具备 Android YAML 自动化执行能力（`midscene`），但脚本创建与执行主要依赖本地文件和命令行，使用门槛较高。需要建设一个可视化页面，支持脚本创建、编辑、AI 生成、执行与报告查看，降低使用成本并提升交付效率。

## 2. 项目目标
1. 提供可视化 YAML 脚本创作能力，支持手动编写与 AI 生成两种方式。
2. 提供一键执行能力，用户可在页面直接触发脚本运行。
3. 提供执行结果与 HTML 报告回显能力，直接复用 Midscene 执行产出的 HTML 报告（如 `shangmenchaoren-login-2026-03-01_18-56-05-2310b71f.html`），不自研报告引擎。
4. 建立最小可用闭环：创建脚本 -> 执行脚本 -> 查看报告。

## 3. 范围定义
### 3.1 In Scope（V1）
1. YAML 脚本管理（新建、编辑、保存、删除、复制）。
2. 两种创建方式。
3. YAML 语法校验与基础规则校验。
4. 脚本执行与执行状态展示。
5. 执行完成后报告查看与下载。
6. 执行历史查询。

## 4. 角色与使用场景
1. 测试/运营人员：通过可视化方式快速创建并执行脚本。
2. 开发人员：用高级语法编辑脚本，验证复杂流程。
3. 场景示例：登录流程自动化、业务关键路径回归、冒烟检查。

## 5. 功能需求
### 5.1 脚本创建方式 A：手动编写 YAML
1. 提供“新建 YAML 脚本”入口。
2. 编辑器支持基础能力：语法高亮、行号、自动缩进、保存。
3. 提供 YAML 模板（含 `android.deviceId`、`tasks.flow` 基础结构）。
4. 点击“校验”时返回语法错误位置（行号、错误原因）。

### 5.2 脚本创建方式 B：自然语言生成 YAML
1. 提供“自然语言输入框”。
2. 用户输入业务描述后，调用 LLM 生成符合规则的 YAML。
3. 生成结果进入编辑器，允许用户二次修改。
4. 生成失败时需返回错误原因，并给出可重试入口。
5. 提供可配置项：设备 ID、默认模型、生成语言（中/英）。

### 5.3 脚本管理
1. 脚本字段：脚本名、创建时间、更新时间、内容、来源（手写/AI 生成）。
2. 支持“另存为”。
3. 支持版本快照（至少保留最近 10 个版本）。

### 5.4 脚本执行
1. 在脚本详情页支持“执行”按钮。
2. 执行前进行校验，校验失败禁止执行并提示。
3. 执行时展示状态机：排队中、执行中、成功、失败。
4. 展示实时日志或阶段日志（最少展示关键步骤与错误信息）。
5. 支持手动终止执行。

### 5.5 报告与结果
1. 直接读取 Midscene 执行产出的报告文件，不自研报告生成逻辑。
2. 执行完成后返回报告路径与访问链接。
3. 支持内嵌预览 HTML 报告。
4. 支持下载报告文件。
5. 历史执行记录可查看对应报告。
6. 结果字段至少包含：开始时间、结束时间、耗时、成功/失败、失败原因。

## 6. 非功能需求
1. 易用性：新用户 10 分钟内可完成首次脚本执行。
2. 性能：脚本保存响应 < 1s（不含网络异常）。
3. 稳定性：执行任务状态不可丢失，页面刷新后可恢复最近状态。
4. 安全性：API Key 等敏感信息不在前端明文存储。
5. 可维护性：前后端接口文档化，错误码标准化。

## 7. 业务流程
1. 用户进入“脚本管理”页面。
2. 选择“手动新建”或“AI 生成”。
3. 完成脚本编辑后点击“保存”。
4. 点击“执行”，系统完成校验并启动执行。
5. 用户查看实时状态。
6. 执行结束后查看 HTML 报告。

## 8. 页面与交互需求
1. 脚本列表页：展示脚本名称、更新时间、来源、最近执行状态。
2. 脚本编辑页：左侧参数区（设备 ID、模型配置），右侧 YAML 编辑器。
3. AI 生成弹窗/面板：输入自然语言 -> 生成 YAML -> 回填编辑器。
4. 执行结果页：日志区 + 报告预览区 + 历史记录。

## 9. 接口需求（建议）
1. `POST /api/scripts`：创建脚本。
2. `PUT /api/scripts/{id}`：更新脚本。
3. `GET /api/scripts`：查询脚本列表。
4. `POST /api/scripts/generate`：自然语言生成 YAML。
5. `POST /api/runs`：发起执行。
6. `GET /api/runs/{id}`：查询执行状态与日志。
7. `GET /api/runs/{id}/report`：获取报告链接或文件。

## 10. 数据模型（建议）
1. Script：`id`、`name`、`content`、`sourceType`、`createdAt`、`updatedAt`。
2. Run：`id`、`scriptId`、`status`、`startedAt`、`endedAt`、`duration`、`log`、`reportPath`、`errorMessage`。
3. ScriptVersion：`id`、`scriptId`、`content`、`versionNo`、`createdAt`。

## 11. 验收标准（V1）
1. 手动创建 YAML 并保存成功。
2. 自然语言可生成可执行 YAML，且可回填编辑器。
3. 脚本可在页面一键执行并返回状态。
4. 执行完成后可查看并下载 HTML 报告。
5. 失败任务能显示可读的错误信息和日志。

## 12. 风险与依赖
1. 依赖本地/远端 Android 设备连接稳定性（ADB）。
2. 依赖 LLM 输出质量，可能出现 YAML 结构不合法。
3. 报告文件存储与访问权限需统一策略。
4. 长任务执行可能引发前端超时，需要异步轮询机制。

## 13. 里程碑建议
1. M1（1 周）：脚本管理 + 手动编辑 + 本地校验。
2. M2（1 周）：执行引擎接入 + 状态回显 + 日志。
3. M3（1 周）：AI 生成 YAML + 报告预览下载。
4. M4（0.5 周）：联调、验收、文档补齐。

## 14. 后续迭代方向（V2+）
1. 多设备并发执行与设备池。
2. 团队协作与权限控制。
3. 脚本调度（定时执行、CI 集成）。
4. 报告聚合看板与失败原因分类。
