# v1.3.0 技术实现计划（plan）

## 1. 计划目标
在现有“场景可按顺序关联多个脚本”的基础上，继续提升为“场景可按任务级进行执行编排”。

本期目标不是简单顺序执行脚本 A -> B -> C，而是支持从已关联脚本中提取 `tasks[]` 下的任务片段，并在场景中重新组合、排序、预览和执行。

交付结果：
- 场景详情页新增“执行编排”区域。
- 可从已关联脚本中选择 `tasks[].name` 加入场景编排列表。
- 编排列表支持调整顺序、移除、备注。
- 场景可预览由多个任务片段拼装而成的最终 YAML。
- 场景可执行整个编排后的脚本。

## 2. 技术方案落地原则
- 优先复用现有场景能力：不推翻“场景关联脚本”现状，而是在其上叠加“任务级编排”。
- 优先可追溯：场景编排不能依赖脚本当前内容漂移，需保存任务快照。
- 优先最小闭环：先支持线性任务编排，不引入条件分支、并行执行、变量替换。
- 保持兼容：现有脚本单独执行、场景脚本顺序维护、运行详情等功能不回归。

## 3. 核心设计思路

### 3.1 现状与目标差异
当前现状：
- 场景支持关联多个脚本
- 关联脚本支持 `sort_order`
- 本质是“脚本级顺序组织”

目标能力：
- 从场景已关联脚本中提取 YAML 里的 `tasks[]`
- 用户按任务粒度选择、排序、组合
- 最终形成一个“场景执行脚本”

也就是：
- 现状：`Scene -> ordered scripts`
- 目标：`Scene -> ordered task snapshots`

### 3.2 为什么不能只保存 task.name
如果只保存 `task.name`，会有以下问题：
- 同一个脚本中可能出现重名 task
- 脚本后续修改后，`task.name` 对应内容可能变化
- 场景执行结果不可追溯

因此建议保存任务快照，而不是只保存 task 名称引用。

## 4. 分阶段实施

## Phase 1：后端任务编排模型与解析能力

### 4.1 目标
为场景新增“任务级编排”数据结构，并提供脚本任务解析能力。

### 4.2 数据模型设计
建议新增表：

1. `scene_task_items`
- `id`
- `scene_id`
- `script_id`
- `scene_script_id`（可选，便于与场景已关联脚本对应）
- `task_index`
- `task_name_snapshot`
- `task_content_snapshot`
- `sort_order`
- `remark`
- `created_at`

设计说明：
- `task_index` 用于识别任务在原脚本中的位置
- `task_name_snapshot` 用于展示
- `task_content_snapshot` 用于生成最终场景 YAML，避免后续脚本修改影响历史编排
- `sort_order` 决定场景执行顺序

### 4.3 脚本任务解析能力
后端需新增一个“解析脚本任务列表”的能力：
- 输入：脚本 YAML
- 输出：脚本的 `tasks[]`
- 返回字段建议：
  - `task_index`
  - `task_name`
  - `flow`
  - `continue_on_error`

建议接口：
- `GET /api/scripts/{script_id}/tasks`

### 4.4 任务编排 API 建议
1. `GET /api/scenes/{scene_id}/task-items`
- 返回场景当前编排列表

2. `POST /api/scenes/{scene_id}/task-items`
- 向场景编排中添加任务项
- 入参：
  - `script_id`
  - `task_index`
  - `remark`

3. `PUT /api/scenes/{scene_id}/task-items/{item_id}`
- 更新编排项
- 支持：
  - `sort_order`
  - `remark`

4. `DELETE /api/scenes/{scene_id}/task-items/{item_id}`
- 删除编排项

5. `GET /api/scenes/{scene_id}/compiled-script`
- 返回场景最终拼装后的 YAML

6. `POST /api/scenes/{scene_id}/runs`
- 直接执行场景拼装脚本

### 4.5 编译策略
将场景中的任务快照按 `sort_order` 拼接为最终 YAML：

```yaml
android:
  deviceId: ""

tasks:
  - name: 前往服务
    flow:
      ...

  - name: 点击预约点签到
    flow:
      ...
```

建议默认策略：
- 环境头优先取场景内第一个任务来源脚本的 `android` 配置
- 如多个脚本头部配置冲突，首期先采用“首个脚本优先 + 提示”

### 4.6 Phase 1 验收
- 后端可解析脚本任务列表
- 场景可保存任务级编排项
- 可生成最终 YAML 预览结果

## Phase 2：前端场景详情页新增“执行编排”

### 5.1 目标
在现有“关联脚本”下方新增“执行编排”区域，完成任务级编排交互。

### 5.2 页面结构建议
场景详情页分为三块：

1. 基本信息
- 场景名称
- 场景说明
- 来源

2. 关联脚本
- 保留现有能力

3. 执行编排
- 左侧：已关联脚本的任务列表
- 右侧：当前场景编排列表

### 5.3 编排交互建议
从“已关联脚本”中：
- 展开查看脚本下的 `tasks[].name`
- 选择任务加入编排

在“当前编排列表”中：
- 展示顺序、任务名、来源脚本、备注、操作
- 支持上移 / 下移 / 删除
- 支持填写备注

### 5.4 编排预览与执行
新增两个入口：
- `预览场景脚本`
- `执行整个场景`

预览：
- 弹窗展示合成后的 YAML

执行：
- 调用场景运行接口
- 运行详情页按 run 维度查看

### 5.5 影响文件（frontend）
- `frontend/src/pages/SceneDetail.vue`
- `frontend/src/api/scenes.ts`
- 可选新增：
  - `frontend/src/components/SceneTaskPicker.vue`
  - `frontend/src/components/SceneCompiledScriptPreview.vue`

### 5.6 Phase 2 验收
- 用户可从已关联脚本中选择 task 加入场景编排
- 编排项顺序可调整
- 可预览最终场景 YAML
- 可发起场景执行

## Phase 3：场景执行链路接入运行系统

### 6.1 目标
将“场景编排后生成的脚本”接入现有运行系统。

### 6.2 实现建议
方案建议：
- 场景执行时动态生成一份临时 YAML
- 复用现有 `runs` 执行链路
- 在 run 上记录：
  - `scene_id`
  - `scene_name_snapshot`
  - `compiled_script_snapshot`

首期不建议做：
- `scene_run_items`
- 子任务级执行跟踪

原因：
- 先用现有 run 模型跑通场景执行闭环
- 降低首期复杂度

### 6.3 运行详情页联动建议
- 运行详情页展示“来源场景”
- 支持查看“本次执行的编排脚本快照”
- 历史场景执行可追溯

### 6.4 Phase 3 验收
- 场景可成功发起执行
- 运行详情可查看场景执行快照
- 不影响现有脚本执行能力

## 8. 风险补充与决策点
- 决策点 1：任务快照是否首期必须落库
  - 建议：必须，避免场景执行结果漂移
- 决策点 2：场景执行是否直接复用现有 run
  - 建议：首期复用，减少模型膨胀
- 决策点 3：多个脚本头部配置冲突如何处理
  - 建议：首期采用首个脚本优先，并在预览时提示
- 决策点 4：是否支持条件分支/失败继续
  - 建议：首期不做，只支持线性编排

## 9. 完成定义（DoD）
- 场景支持按任务级进行执行编排
- 可从关联脚本中挑选任务片段加入场景
- 可调整顺序并预览最终场景 YAML
- 可执行整个场景编排结果
- 编排结果具备可追溯性
