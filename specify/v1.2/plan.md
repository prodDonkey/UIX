# v1.2.0 技术实现计划（plan）

## 1. 计划目标
基于现有“脚本列表 / 运行列表”能力，新增“场景列表”模块，实现“一个场景可关联多个脚本”的组织管理能力，并保持现有脚本执行链路稳定。

交付结果：
- 前端新增“场景列表”入口与场景详情页。
- 后端新增场景与场景脚本关联模型及 CRUD API。
- 场景内可维护脚本关联、顺序和备注。
- 现有脚本管理、脚本运行、运行列表功能保持兼容。

## 2. 技术方案落地原则
- 优先最小改动：不重构现有脚本执行链路，先补齐场景管理能力。
- 保持兼容：已有脚本 CRUD、运行接口、运行详情页面不破坏。
- 优先复用：场景只做组织层，底层继续复用现有脚本资产。
- 可扩展：为后续“场景运行”预留数据结构，但本期不强行落地复杂编排。

## 3. 分阶段实施

## Phase 1：后端数据模型与 API 落地
### 3.1 目标
新增场景实体与场景脚本关联实体，提供基础 CRUD 与关联维护能力。

### 3.2 数据模型设计
建议新增表：

1. `scenes`
- `id`
- `name`
- `description`
- `source_type`
- `created_at`
- `updated_at`

2. `scene_scripts`
- `id`
- `scene_id`
- `script_id`
- `sort_order`
- `remark`
- `created_at`

设计说明：
- `scene_scripts` 作为关联表，支持一个场景关联多个脚本，也支持一个脚本被多个场景复用。
- `sort_order` 用于场景内脚本排序，便于后续扩展顺序执行。
- `remark` 用于记录该脚本在场景下的用途说明，避免用户只看脚本名难以理解。

### 3.3 API 建议
1. `GET /api/scenes`
- 返回场景列表
- 字段建议包含：`id/name/source_type/updated_at/script_count`

2. `POST /api/scenes`
- 创建场景
- 入参：`name/description/source_type`

3. `GET /api/scenes/{scene_id}`
- 返回场景详情
- 包含场景基本信息与关联脚本列表

4. `PUT /api/scenes/{scene_id}`
- 更新场景基本信息

5. `DELETE /api/scenes/{scene_id}`
- 删除场景

6. `POST /api/scenes/{scene_id}/scripts`
- 向场景中添加脚本
- 入参：`script_id/sort_order/remark`

7. `PUT /api/scenes/{scene_id}/scripts/{relation_id}`
- 更新关联项
- 支持修改 `sort_order/remark`

8. `DELETE /api/scenes/{scene_id}/scripts/{relation_id}`
- 删除场景中的脚本关联

### 3.4 影响文件（backend）
- `app/models/scene.py`
- `app/models/scene_script.py`
- `app/models/__init__.py`
- `app/schemas/scene.py`
- `app/api/scenes.py`
- `app/main.py`

必要时补充：
- `app/services/scene_service.py`

### 3.5 迁移策略
- 若当前环境主要依赖 `create_all`，先按现有模式新增表，避免引入额外迁移复杂度。
- 若后续进入稳定生产阶段，建议补充 Alembic migration，显式管理新增表结构。

### 3.6 Phase 1 验收
- 可创建、编辑、删除场景。
- 可为场景添加多个脚本并保存顺序。
- 删除场景不影响脚本资产本身。
- 已有关联脚本可被多个场景复用。

## Phase 2：前端场景管理页面落地
### 4.1 目标
在现有顶部导航中新增“场景列表”，实现完整的场景管理 UI。

### 4.2 页面与路由建议
新增页面：
- `src/pages/SceneList.vue`
- `src/pages/SceneDetail.vue`

新增路由：
- `/scenes`
- `/scenes/:id`

导航调整：
- 顶部导航改为：`场景列表 / 脚本列表 / 运行列表`
- 默认首页建议进入 `场景列表`

### 4.3 场景列表页设计
表格字段建议：
- 场景名称
- 脚本数
- 来源
- 更新时间
- 操作

操作建议：
- 查看/编辑
- 复制（可后置）
- 删除

顶部操作：
- `新建场景`

### 4.4 场景详情页设计
页面拆分为两块：

1. 基本信息
- 场景名称
- 场景说明
- 来源

2. 关联脚本
- 列表字段：顺序、脚本名称、来源、更新时间、备注、操作
- 操作：添加脚本、移除脚本、调整顺序、编辑备注

交互建议：
- “添加脚本”采用弹窗选择已有脚本
- 默认只复用已有脚本，不在该页面直接创建新脚本
- 顺序调整 MVP 可先用“上移/下移”，后续再升级拖拽

### 4.5 影响文件（frontend）
- `src/App.vue`
- `src/router/index.ts` 或对应路由文件
- `src/api/scenes.ts`
- `src/pages/SceneList.vue`
- `src/pages/SceneDetail.vue`

可选补充：
- `src/components/SceneScriptPicker.vue`

### 4.6 Phase 2 验收
- 页面可正常展示场景列表与详情。
- 可在场景中添加、删除、排序脚本。
- 不影响现有脚本列表与运行列表访问路径。

## Phase 3：脚本资产与场景的联动优化
### 5.1 目标
补齐场景视角和脚本视角之间的基础联动，提升可管理性。

### 5.2 优化项建议
- 在脚本列表中增加“所属场景数”列。
- 脚本详情页可展示“被哪些场景引用”。
- 删除脚本时，若存在场景引用，给出明确提示。

### 5.3 兼容性处理
- 若脚本被场景引用，删除脚本时应同步删除关联表记录，或显式阻止删除。
- 建议本期采用“允许删除，但二次确认提示影响范围”，实现成本更低。

### 5.4 Phase 3 验收
- 用户能快速判断脚本是否被场景使用。
- 删除行为具备可理解的风险提示。

## Phase 4：为后续场景运行预留扩展
### 6.1 目标
本期不实现完整“场景运行”，但在设计上为后续顺序执行多个脚本留边界。

### 6.2 预留方向
- `scene_scripts.sort_order` 作为未来执行顺序依据。
- 场景详情页保留“运行场景”按钮位置，但本期可隐藏或标注“规划中”。
- 后续如需实现，可新增：
  - `scene_runs`
  - `scene_run_items`

### 6.3 本期范围控制
- 本期不改造 `runs` 主模型，不把 run 直接绑定到 scene。
- 本期继续沿用“脚本运行”模式，避免扩大发版风险。

## 10. 风险补充与决策点
- 决策点 1：脚本与场景关系是否支持多对多
  - 建议：直接采用多对多，避免后续数据模型返工。
- 决策点 2：场景内排序是否首期支持拖拽
  - 建议：MVP 先用上移/下移，减少交互与状态复杂度。
- 决策点 3：脚本删除时如何处理被引用场景
  - 建议：首期采用强提示 + 允许删除，兼顾实现成本与可用性。

## 11. 完成定义（DoD）
- 功能：场景可创建、编辑、删除，并可关联多个脚本。
- 兼容：现有脚本管理与运行功能无回归。
- 可用：用户能从场景视角组织和维护脚本资产。
- 文档：`specify/v1.2/plan.md` 完成并可作为开发依据。
